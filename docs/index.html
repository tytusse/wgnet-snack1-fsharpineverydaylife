## F# w życiu codziennym Przykład działającej "grubej" aplikacji Notes: - Przedstawić się: - programista .NET od 10 lat, doświadczenie od .NET 1.1, trochę programowania web, ASP.NET MVC, trochę innego :) - obecnie na garnuszku prywatnym, z ograniczoną odpowiedzialnością, projekty backendowy oraz trochę webowych - Spytać kto używa F# i gdzie
## O czym by tu ... - struktura projektu - projects / modules / classes / DI / paket / tests - async + MailboxProcessor + Result + computation expressions + type providers + ... - use + IDisposable + inline object - testy - type inference / tuples / custom operators - F# interactive / scripting / fsx - o tooling'u ... - skąd F#?
## Aplikacja kod ⇒ Visual Studio Notes: - pokazać projekty, omówić krótko w kilku słowach - Core - może być współdzielona, boilerplate tu, - Service - konkretna aplikacja używająca Core - samo mięcho - unit testy - (????) system testy - automatyczne testy "integracyjne" - moduły, klasy i serwisy (na podstawie CandyService) - klasy w modułąch - unikamy namespace - unikamy otwierania modułów - używamy aliasów by nadać znaczenie primitywom - DI - Windsor - rozszeżenie do FSharp Option - testy - pokazać testy, nazwy plików - module level vs. class level - pokazać widok R# / uruchomić + debug mode - tooling - pokazać szukanie w VS2017 (ctrl+shift+g) - pokazac refactor-rename (ctrl+r+r)
## Tooling ... - podstawowe funkcje działają: podświetlanie typów (ważne przy type inference), rename, ... - zdecydowanie uboższy niż dla C#, szczególnie boli słabszy intelisense - nie działa z R#, choć unit testy do pewnego stopnia obsługuje - treat **FS0025** compiler warning as error
## Uwagi końcowe - F# jest też językiem obiektowym - nie należy się bać OOP - F# może wyrazić większość rzeczy z C# (w tym na przykład P/Invoke) - F# może współpracować z C# (extension methods, default parameters, value tuple, ...) - FSX + VS Code = środowisko skryptowe, playground - Tooling w VS dla F# jest uboższy niż np. dla C#
